Course Framing

The Core Idea

Most financial questions aren’t about the “right” answer.

They’re about making the best decision
given noise, uncertainty, and tradeoffs.

This is the thesis of the course. Pause after the first line. Emphasize that finance lives in uncertainty, not closed-form answers. Use this to pivot from “who I am” to “why this course is structured the way it is.”

What This Course Is — And Isn’t

This course is not - learning every algorithm - optimizing models for their own sake

This course is - applied statistical learning for finance - regression and classification - interpretation and model assessment - understanding tradeoffs (accuracy vs robustness, simplicity vs flexibility)

Stress that depth of understanding matters more than breadth. This sets expectations early and lowers anxiety.

Tools And Transfer

The ideas matter more than the language.

  • Part 1 uses R (lecture language)
  • Part 2 uses Python
  • Assignments may be completed in either language
  • The workflow and concepts transfer directly across both

Reinforce portability: if they understand the workflow here, Python later should feel familiar rather than intimidating.

The Data Science And Modeling Workflow

(Finance-First)

The Workflow We Will Use All Semester

This is not a one-time sequence.
It’s a loop you will revisit constantly.

This is the backbone of the course. Everything we do in R this semester fits somewhere in this loop. Emphasize that we will cycle through it repeatedly, not march through it once.

Why This Is A Loop (Not A Checklist)

  • New data arrives
  • Assumptions change
  • Results raise new questions
  • Decisions require iteration

In finance, the workflow rarely ends.

Stress iteration and feedback. Connect to real-world finance: markets move, regimes change, models decay. This motivates reproducibility and disciplined workflows.

What Each Step Looks Like In Finance

  • Import
    Prices, returns, fundamentals, loan tapes, macro series

  • Tidy
    Dates, joins, missing data, panel structure

  • Explore
    Distributions, correlations, regime changes

  • Model
    Regression and classification to support decisions

  • Communicate
    Plots, narrative, and a clear “so what?”

Walk through quickly. Reinforce that modeling is only one part of the workflow. Communication is where analysis becomes a decision.

R And Python: Portability

The Key Claim

If you understand what we’re doing in R,
you’ll recognize it instantly in Python.

Say this slowly and confidently. The goal is to reduce anxiety about the second half of the course. Emphasize that we are teaching ideas and workflows, not syntax trivia.

What Actually Transfers

  • Data frames / tables
  • Functions, objects, and libraries
  • Wrangle → visualize → model → evaluate
  • Train / test thinking (coming soon)

Point out that the verbs and structure stay the same. R syntax changes to Python syntax, but the workflow does not. This sets up Part 2 cleanly.

RStudio Orientation

(Minimum Viable Comfort)

R vs RStudio

  • R = the engine
  • RStudio = the interface

Keep this high level. R is the language and computation engine. RStudio is the environment that makes R usable: editor, console, plots, files, help. Do not list features yet — this is orientation, not a tour.

Packages

Use the phone analogy here.

R by itself is like a new phone: - it turns on - it can do a few basic things - but it isn’t very useful yet

Packages are the apps you install on the phone. install.packages() is downloading an app. library() is opening the app.

Most of what we do this semester is learning: - which “apps” to use - and when to use them.

Projects (Strongly Recommended)

  • One folder per project
  • Relative paths
  • Easier reproducibility

Emphasize that projects prevent path problems. This sets up why data lives in data/ and why here::here() exists later. Students don’t need to understand this deeply yet — just accept it.

RStudio Walkthrough (Live)

This is a live, interactive walkthrough. Show the four panes: - Source (scripts) - Console (where code runs) - Environment / History - Files / Plots / Packages / Help

Demonstrate: - creating a new script - running a single line (Cmd/Ctrl + Enter) - running multiple lines - clearing the console - where plots appear - where errors show up

Keep this informal. Encourage students to follow along. Reassure them that it’s normal to feel lost at first.
